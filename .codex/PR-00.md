# PR-00.md — Greentic Integration Tester/Validator: Repo scaffolding + CI

## Repo
You are working in repo: **greentic-integration**

## Goal
Create the initial workspace and engineering scaffolding for a CLI-driven integration test framework:
- `greentic-integration-tester` — executes `.gtest` scripts (commands + directives)
- `greentic-integration-validator` — assertions used *inside* `.gtest` scripts (filesystem, json, cbor presence checks)

This PR focuses on:
- crate layout
- shared `core` crate for parsing + substitution
- CI + local checks
- minimal docs and examples folder layout

## Non-goals
- Implementing full runner/validator functionality (that’s PR-01+).
- Any coupling to Greentic internal type schemas.
- Networked tests by default; tests should run offline unless explicitly configured.

## Deliverables
1) Workspace layout:
```
greentic-integration/
  Cargo.toml
  crates/
    greentic-integration-core/
    greentic-integration-tester/
    greentic-integration-validator/
  tests/
    gtests/
    fixtures/
  docs/
  .codex/
  .github/workflows/
  scripts/
```

2) `.codex/` standard files:
- `.codex/global_rules.md`
- `.codex/repo_overview.md`
- `.codex/STATE.json` (minimal; used by your automation, keep small)
- `scripts/local_check.sh` to run fmt/clippy/test

3) CI workflow (GitHub Actions):
- On push/PR: `cargo fmt --check`, `cargo clippy --all-targets --all-features -D warnings`, `cargo test --all-features`
- Cache cargo
- Run on ubuntu-latest

4) Crate skeletons:
- Each crate compiles with a minimal `main.rs` or `lib.rs`
- `core` exposes modules: `parse`, `substitute`, `model`, `errors`
- Binaries have clap-based skeleton + `--help` works

## Dependencies
Use:
- `clap = { version = "4", features = ["derive"] }`
- `anyhow`, `thiserror`
- `serde`, `serde_json`
- Keep dependencies conservative.

## Implementation Steps
### 1) Workspace `Cargo.toml`
- Create a workspace with members:
  - `crates/greentic-integration-core`
  - `crates/greentic-integration-tester`
  - `crates/greentic-integration-validator`
- Set common package metadata in `[workspace.package]` where appropriate.
- Enable resolver = "2".

### 2) Create `greentic-integration-core`
- `src/lib.rs` exports modules:
  - `pub mod model;`
  - `pub mod parse;`
  - `pub mod substitute;`
  - `pub mod errors;`
- Provide placeholder structs and doc-comments for:
  - `Directive`, `CommandLine`, `Step`, `TestPlan`
  - `SubstitutionContext`
- Include unit test skeleton.

### 3) Create binaries
#### `greentic-integration-tester`
- `src/main.rs`:
  - clap struct with `--test <path>` and `--workdir <path>` (optional)
  - For now, just prints parsed steps (stub), exits 0.
- Must build and run `--help`.

#### `greentic-integration-validator`
- `src/main.rs`:
  - clap struct with subcommands stub:
    - `file exists <path>`
    - `file not-exists <path>`
  - For now, implement only these two checks.
  - Return exit 0 on pass, exit 2 on validation fail, exit 1 on internal error.

### 4) Scripts
Create `scripts/local_check.sh`:
- `set -euo pipefail`
- run `cargo fmt --all --check`
- run `cargo clippy --all-targets --all-features -- -D warnings`
- run `cargo test --all-features`

### 5) CI
Add `.github/workflows/ci.yml` with:
- trigger: push, pull_request
- steps: checkout, toolchain stable, cache, run `scripts/local_check.sh`

### 6) Docs
Add `docs/README.md` describing:
- What `.gtest` is (one command per line + directives)
- How validator is used inside scripts
- Where examples live

## Acceptance Criteria
- `cargo build` succeeds
- `scripts/local_check.sh` succeeds locally
- CI passes
- `greentic-integration-tester --help` and `greentic-integration-validator --help` work
- `greentic-integration-validator file exists Cargo.toml` passes in repo root
