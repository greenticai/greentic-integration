# PR-02.md — Tester: execute steps, workdir isolation, captures, reporting

## Repo
You are working in repo: **greentic-integration**

## Goal
Implement `greentic-integration-tester` as the `.gtest` runner:
- Executes each command line (argv) with OS process spawn (no shell)
- Applies directives to execution context
- Captures outputs when requested
- Produces readable transcript and a machine-readable JSON report

## CLI
`greentic-integration-tester`:
- `--test <path>` (file or directory; if dir, discover `**/*.gtest`)
- `--workdir <path>` (optional; if omitted, create a temp per test)
- `--keep-workdir` (don’t delete temp on success)
- `--repo-root <path>` (optional; default = current dir)
- `--prepend-path <path1[:path2...]>` (optional; prepended to PATH for spawned processes)
- `--fail-fast`
- `--concurrency <N>` (optional; default 1)
- `--report <text|json>` (default text)
- `--report-file <path>` (optional; if set, write report there)

## Execution semantics
### Per-test initialization
- Determine `TEST_DIR` = directory containing the `.gtest` file
- Create `WORK_DIR`:
  - If `--workdir` provided: use `<workdir>/<test_name_sanitized>` (create)
  - Else: create temp dir (e.g., in OS temp) with unique name
- Built-in substitution vars at start:
  - `TEST_DIR`, `WORK_DIR`, `REPO_ROOT`, `TMP_DIR` (TMP_DIR = WORK_DIR/tmp created)
- Start `SubstitutionContext` empty; as directives occur, update test_vars/env_vars.

### Directives effect
- `@set`: updates `test_vars`
- `@env`: updates env override map for future commands (also available for substitution)
- `@cd`: changes current working directory for future commands (relative to current cwd)
- `@timeout`: sets default timeout for future commands (until changed)
- `@expect`: sets expectation for *next command only*, then resets to default (exit=0)
- `@capture NAME`: applies to *next command only*; store captured stdout/stderr/exit in capture store by NAME
- `@print NAME`: prints stored capture to stdout (useful debug). If missing, error.

### Commands execution
- Token argv already comes from parser.
- Apply substitution to each token and to cwd/env values.
- Spawn with:
  - `std::process::Command`
  - `current_dir` from context
  - environment = OS env + overrides from `@env` + PATH prepend if requested
- Capture stdout/stderr always internally with bounded buffers (e.g., 1MB each) to show transcript; but store full (bounded) in report.
- Enforce timeout:
  - Use a cross-platform approach:
    - simplest: `wait_timeout` crate OR implement with `std::thread` + kill (platform specific)
  - Prefer adding dependency `wait-timeout = "0.2"` (small, common).
- If timeout triggers: kill process, mark step failed with `TimedOut`.

### Expectations
- Default expectation: exit code == 0
- If `@expect exit!=0`, then any non-zero exit passes (but still record exit code)
- If expectation fails: stop test (unless you implement `--continue-on-fail`, not required)
- `@expect` resets after one command.

## Captures
Store:
- name
- argv
- cwd
- exit_code
- stdout (string, lossy UTF-8 ok)
- stderr
- duration_ms
- timed_out bool

When `@capture NAME` is used, store the capture for the next command under that name.
If the same NAME is reused, overwrite (document this).

## Reporting
### Text transcript
For each command:
- Print:
  - `[$line_no] $ <argv...>`
  - `cwd: ...` (only if changed or verbose)
  - `exit: N` and duration
  - stdout/stderr truncated (e.g. 8k each), with markers when truncated
On failure: print a final summary and path to workdir.

### JSON report
Write a JSON object containing:
- test path
- workdir path
- start/end timestamp
- overall status
- steps[] with:
  - line_no
  - raw
  - kind (directive/command)
  - for command: argv, cwd, env_overrides (keys only), exit, stdout, stderr, duration_ms, timed_out
  - for directive: parsed fields
- captures map keyed by name with captured command results

Add a `Report` struct in `tester` crate (or core) with `serde::Serialize`.

## Parallel execution
If `--test` is a directory and `--concurrency > 1`:
- run tests in parallel using a thread pool (e.g., `rayon` or `std::thread` + channel)
- ensure each test has its own workdir
- output:
  - either interleaved logs (ok) or buffered per test (better). Keep simple: buffered per test then print.
- If `--fail-fast`: stop scheduling new tests after first failure; allow running tests to finish.

## Acceptance Criteria
- Running a simple `.gtest` that calls `echo` (or `rustc --version`) works
- Directives apply correctly
- `@capture` + `@print` works
- JSON report can be emitted and includes step results
- Timeouts kill long-running commands
- Helpful error messages include line numbers and the raw line

## Notes
- Do not run commands through a shell.
- Be careful with PATH prepend: use platform separator (`:` on unix, `;` on windows).
- Redact secrets in transcript:
  - If env key matches `*_TOKEN`, `*_SECRET`, `*_PASSWORD`, do not print values.
