## Example `.gtest` content guidance

### 00_smoke_validator.gtest
- Use `@set` and `${WORK_DIR}`
- Create a json file via a small inline command:
  - easiest cross-platform is to rely on `python` may not exist. Prefer to create via `greentic-integration-validator`? Not its job.
  - Alternative: include `tests/fixtures/sample.json` and copy it.
- Copy fixture into `${WORK_DIR}` (use `cp` may be non-windows). To keep cross-platform, do not rely on shell utilities.
Instead:
- Add a tiny helper subcommand to tester like `@writefile path "content"`? Avoid scope creep.
So prefer committing fixtures and using validator on them directly in `${TEST_DIR}`.

Example steps:
- `greentic-integration-validator json path ${TEST_DIR}/../fixtures/sample.json a.b[0].c --eq 3`
- `greentic-integration-validator cbor find ${TEST_DIR}/../fixtures/sample.cbor --string "hello"`

### 01_capture_and_expect.gtest
- Run a command expected to fail:
  - Use the validator itself on a missing file:
    - `@expect exit!=0`
    - `@capture missing`
    - `greentic-integration-validator file exists ${WORK_DIR}/nope.txt`
  - Then assert captured stderr contains something (needs capture-output validation in runner OR just `@print` for demo)
If runner doesn’t support capture-validation, at least show capture output.
